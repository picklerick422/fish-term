# fish-term 首字符不显示 Bug — 调查与修复记录

> 平台：HarmonyOS 终端 App，XComponent + NativeWindow 自绘渲染  
> 渲染引擎：libghostty-vt + OH_Drawing  
> 相关代码：`libghostty_ohos/src/main/cpp/`、`libghostty_ohos/src/main/ets/`

---

## 1. 问题现象

- 用外接物理键盘输入时，按下第一个字符后，光标会后移，但光标前面没有显示字符。
- 再按下一个键后，前一个字符才出现，表现为始终滞后一个按键。
- 在 Claude Code 输入框中可稳定复现：输入 `h` 后光标后移一格，但 `h` 所在 cell 是空白。
- 同时观察到输入框最左侧的 `>` 提示符会轻微下移，说明画面不是完全没刷新，而是文本 typography 布局发生了变化。
- 软键盘/触摸场景不明显，容易被额外的 ArkUI 帧刷新掩盖。

典型表现：输入 `hello world` 时，最新输入的字符看起来丢失，下一次输入后才补出来。

---

## 2. 诊断方法

通过 `FT_DIAG` 日志逐段验证链路：

- `terminal.cpp::feedOutput`：确认 shell 回显已经进入终端 core。
- `terminal.cpp::drawFrame`：打印 frame ID、cursor 位置、`leftOfCursorCp`，确认光标左侧 cell 的 codepoint。
- `napi_init.cpp::DispatchKeyEvent`：确认外接键盘输入被转换成正确 byte sequence。
- `napi_init.cpp::OnFrameTick` / NativeVSync：确认 native draw/flush 是否被触发。

关键日志曾显示：

```text
DispatchKeyEvent writeInput seq[0]=104 len=1
feedOutput len=36 first=0x1B
drawFrame frame=397 cursor=(30,3) leftOfCursorCp=0x68
```

这说明：

1. `h` 已经进入 Ghostty grid。
2. cursor 已经后移。
3. native drawFrame 已经看到 `leftOfCursorCp=0x68`。
4. 但屏幕上该 cell 仍为空白。

因此最终问题不在输入、WebSocket、VT 解析，也不在 cell 数据生成，而在 renderer 的文本绘制阶段。

---

## 3. 最终根因

最终有效根因是 **OH_Drawing typography run 的右边界裁剪**。

原来的 `NativeDrawingRenderer::renderGrid()` 会把同一行里连续、同样式的 cell 合并成一个 typography run：

```text
cell0 + cell1 + ... + 最新输入字符
  -> runText
  -> TypographyLayout(maxWidth = runCols * cellWidth)
  -> TypographyPaint(left = startCol * cellWidth)
```

终端 cell 宽度来自探针字符测量，不是每个 glyph 的严格最大 advance/overhang。OH_Drawing 对 typography layout 宽度有裁剪行为。当最新输入字符位于 run 的最右端时，如果实际 glyph advance 或右侧 overhang 稍微超过 `runCols * cellWidth`，该 glyph 就会被裁掉。

下一次按键后，run 变宽，前一个字符不再位于最右边界，于是前一个字符出现，看起来像“首字符滞后一个按键”。

`>` 提示符轻微下移也符合这个判断：合并 run 的内容变化后，OH_Drawing 重新布局整段 typography，导致 baseline/height 产生微小变化。

---

## 4. 正确修复

有效修复在 `libghostty_ohos/src/main/cpp/renderer/native_drawing_renderer.cpp`：

1. 去掉连续 cell 合并成 typography run 的逻辑。
2. 改为逐 cell layout/paint。
3. 对单 glyph 的 typography layout 宽度增加少量 overhang 余量。

修复后的关键策略：

```cpp
GlyphLayout* layout = getGlyphLayout(firstText, visualAttrs, span);
if (layout && layout->typography) {
    const float left = col * cellWidth;
    const float top = row * cellHeight;
    const float y = top + std::max(0.0f, (cellHeight - layout->height) * 0.5f);
    OH_Drawing_TypographyPaint(layout->typography, m_canvas, left, y);
}
col += span;
```

并在 `getGlyphLayout()` 中给 layout 宽度留出余量：

```cpp
const double overhang = std::max(2.0, static_cast<double>(m_fontSize * m_density) * 0.25);
const double maxWidth = std::max(1.0, static_cast<double>(m_cellWidth * span) + overhang);
OH_Drawing_TypographyLayout(typography, maxWidth);
```

这样最新输入的字符不再依赖“下一个字符扩大 run width”才能显示，也避免 run 级 typography 重新布局造成的提示符轻微跳动。

验证结果：外接键盘在 Claude Code 输入框中输入单个 `h` 后，字符立即显示，功能恢复正常。

---

## 5. 曾经误判的呈现链路问题

排查过程中曾经认为根因是：

> native buffer 已经 flush，但 ArkUI/RenderService 没有合成窗口帧。

这个方向有一定背景价值，但不是本问题的最终根因。

相关尝试：

- `OH_NativeVSync` 驱动 native draw/flush。
- native flush 后置位 `m_framePresented`。
- ArkTS 侧 16ms 轮询 `drainPendingFrame()`。
- bump `@State framePresentTick` 强制 ArkUI 提交窗口帧。
- 将 XComponent 从 `SURFACE` 改为 `TEXTURE`，并把 `framePresentTick` 绑定到 XComponent 自身的 `backgroundColor`，而不是隐藏兄弟节点。

这些改动可以改善“native buffer 没有被 latch/采样”的风险，但用户后续观察到“光标后移但字符为空白”，说明画面已经刷新，最终问题落在 glyph 绘制而非呈现。

保留的经验：

- 只 dirty 一个 XComponent 的兄弟节点，不一定会重新录制/重新采样 XComponent 自身。
- 若要通过 ArkTS 触发 XComponent 采样，状态变化应绑定在 XComponent 节点自身，例如轻微翻转其 `backgroundColor`。
- `OH_NativeVSync` 能解决“何时绘制”，但不能单独保证 ArkUI/RS 呈现 native buffer。

## 6. 失败尝试记录

| 尝试 | 假设 | 结果 |
|------|------|------|
| IME guard / `m_wantsIme` / preview text | 首字符被 IME 链路吞掉 | 日志证明输入和 VT grid 正常，方向错误 |
| `kBufferUsage` 加 `CPU_READ | MEM_DMA` | buffer 内存不可被 RS 直接扫描 | 无效 |
| 从 `OnFrameTick` 渲染 | ArkUI 帧管线内 flush 会被即时合成 | 空闲时 `OnFrameTick` 不稳定，不解决 |
| 仅用 `OH_NativeVSync` 驱动渲染 | 独立 vsync 会持续合成 | 只能驱动绘制，不能保证呈现 |
| 隐藏 1px `Text` bump `@State` | 提交窗口帧即可 latch buffer | 不保证重新录制 XComponent 节点 |
| 只改 `TEXTURE` + dirty XComponent | 问题是 buffer 采样/latch | 光标会动但字符仍空白，说明不是最终根因 |

## 7. 关键技术点

- 终端渲染不适合把整段 ASCII 合并成普通 typography paragraph。终端 cell 是固定网格，逐 cell 绘制更可靠。
- `cellWidth` 是终端布局单位，不是所有 glyph 的严格裁剪边界。
- OH_Drawing typography layout 宽度可能裁剪最右侧 glyph。
- 最新输入字符位于 run 的最右边界时，最容易被裁掉。
- 给单 glyph layout 留少量 overhang 可以避免右侧裁剪。
- XComponent 呈现问题和 glyph 裁剪问题表象相似，但可通过“光标是否立即后移”区分：
  - 光标不动：优先查呈现/latch。
  - 光标后移但字符空白：优先查 glyph 绘制/layout/clip。
