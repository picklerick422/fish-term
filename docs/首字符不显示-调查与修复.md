# fish-term 首字符不显示 Bug — 调查与修复全记录

> 平台：HarmonyOS（鸿蒙）终端 App，XComponent SURFACE 模式 + NativeWindow 自绘渲染
> 渲染引擎：libghostty-vt（ghostty）+ OH_Drawing（native drawing）
> 相关代码：`libghostty_ohos/src/main/cpp/`、`libghostty_ohos/src/main/ets/`

---

## 1. 问题现象

- 用**外接物理键盘**输入时，按下第一个字符后屏幕**不显示**该字符。
- 必须再按下一个键，前一个字符才出现（始终滞后一个按键）。
- 关键：**干等几秒也不显示**，只有再次按键才触发显示。
- 软键盘/触摸场景不明显（触摸事件会持续唤醒 ArkUI 帧管线，掩盖了问题）。

典型表现：输入 "hello world" 显示成 "hell world"，"/" 在按下下一个键前不显示。

---

## 2. 诊断方法

通过在关键链路插入 `FT_DIAG` 级别 `OH_LOG_INFO` 日志，逐段验证数据流：

- `terminal.cpp::feedOutput` 开头：打印 len、首字节
- `terminal.cpp::drawFrame` 末尾：打印 frame ID、cursor 位置、`leftOfCursorCp`（光标左侧字符码点）
- `napi_init.cpp::OnFrameTick`：渲染心跳、content/blink/trailing 分类
- `napi_init.cpp::DispatchKeyEvent`：按键码、writeInput 序列
- `napi_init.cpp::InsertImeText` 等 IME 链路

## 3. 决定性证据（来自真实 hilog）

按下 'h' 后的空闲期日志（节选，已确认可靠）：

```
16:54:09.854  DispatchKeyEvent code=2024
16:54:09.854  DispatchKeyEvent writeInput seq[0]=104 len=1      # 'h' 写入
16:54:09.858  feedOutput len=36 first=0x1B                       # shell 回显
16:54:09.907  drawFrame frame=397 cursor=(30,3) leftOfCursorCp=0x68   # 'h' 已进入 grid 并绘制
16:54:09.949  drawFrame frame=398 ... leftOfCursorCp=0x68
...
16:54:10.x    drawFrame frame=400..405 ... leftOfCursorCp=0x68   # 持续重绘，'h' 一直在
16:54:08.507  WMSMain  W  OnVsyncTimeOut: in                     # 窗口 vsync 超时休眠
```

**关键结论链：**

1. `'h'` 已进入 grid ✓（`leftOfCursorCp=0x68`）
2. `drawFrame` 被持续调用 ✓（光标闪烁每 250ms 触发）
3. `FlushBuffer` 被调用 ✓（endFrame 无条件 flush）
4. **但屏幕不更新，直到按键** ✗
5. `OnVsyncTimeOut`（WMSMain，主线程）= **窗口/ArkUI 的 vsync 连接休眠了**

> 在引入 OH_NativeVSync 后，`drawFrame` 从 UI 线程(5313) 改到 vsync 回调线程(**8974**)，每个 blink 周期(250ms)及时绘制 —— 但屏幕**仍然**不刷新。这证明问题不在"谁驱动绘制"。

---

## 4. 根本原因（已确认）

> **绘制 ≠ 呈现。**

在 HarmonyOS 中，XComponent SURFACE 图层的 **RenderService（RS）合成是由 ArkUI 提交的"窗口帧"门控的**，而**不是**由我们对 NativeWindow 的 `FlushBuffer` 直接触发的。

事件链：

```
外接键盘输入（无触摸事件）
  → ArkUI 帧管线判定为空闲
  → ArkUI 停止向 RS 请求窗口帧（OnVsyncTimeOut）
  → RS 不对窗口（含我们的 surface 图层）做合成
  → 我们 FlushBuffer 的缓冲区在 BufferQueue 中排队，但得不到合成上屏
  → 直到按下一个键：
       按键事件 → ArkUI 请求窗口帧 → RS 合成整个窗口
       → latch 我们最新排队的缓冲区（含上一个字符）→ 显示
```

这完美解释了"始终滞后一个按键"和"干等不显示、按键才显示"。

## 5. 失败的尝试（记录，避免重走弯路）

| 尝试 | 假设 | 结果 |
|------|------|------|
| IME guard / m_wantsIme / preview text 方向 | 首字符被 IME 链路吞掉 | FT_DIAG 证明数据流正常，方向全错 |
| `kBufferUsage` 加 `CPU_READ \| MEM_DMA`（commit b816562） | 缓冲区在 CPU-only 内存，RS 无法直接 DMA 扫描 | 无效，"还是不行" |
| 从 OnFrameTick（ArkUI 帧回调）渲染 | ArkUI 帧管线内 flush 会被即时合成 | 无效：ArkUI **空闲时根本不调用 OnFrameTick**，内容渲染被推迟到按键 |
| 仅用 OH_NativeVSync 驱动渲染（commit 1a605dd） | 独立 vsync 持续驱动 → RS 持续合成 | **部分有效但不够**：drawFrame 在 vsync 线程及时跑了，但 flush 仍不被 RS 合成（窗口帧门控） |

**教训：** OH_NativeVSync 解决了"何时绘制"，但没解决"何时呈现/合成"。独立 vsync 连接无法唤醒**窗口**的合成。

---

## 6. 最终修复方案

> **每次 native flush 新帧后，主动让 ArkUI 做一次"窗口帧"，复刻按键的呈现路径。**

OH_NativeVSync 负责"何时绘制"；ArkTS 的 `@State` bump 负责"何时唤醒窗口合成"。两者配合，空闲时也能即时上屏。

### 6.1 数据流

```
内容变化 / 光标闪烁 / trailing
  → OH_NativeVSync 回调 OnVsyncTick（线程 8974）
  → DrawFrameLocked() 绘制 + FlushBuffer
  → 置 m_framePresented = true
  → ArkTS 16ms 定时器 drainPendingFrame() 读到 true
  → bump @State framePresentTick
  → 重跑 build() → ArkUI 提交窗口帧
  → RS 合成整个窗口 → 呈现最新缓冲区
```

### 6.2 改动清单

**C++ (`napi_init.cpp`)**

- 新增 `OH_NativeVSync* m_nativeVSync` + `std::atomic<bool> m_vsyncPending`，用 `OH_NativeVSync_Create/RequestFrame/Destroy` 驱动渲染。
- `RequestVsyncFrame()`：用 `compare_exchange` 去重，避免重复请求。
- `OnVsyncTick()`：分两阶段加锁——
  - 阶段1（`m_renderMutex`）：消费 `m_renderDirty` / `m_trailingFrames`。
  - 阶段2（`m_surfaceMutex`）：检查光标闪烁 + 调用 `m_terminal->drawFrame()`（`m_renderer`/`m_lastBlinkMs` 必须在 `m_surfaceMutex` 下访问）。
  - 还需渲染（闪烁开启 / 有 trailing / 仍 dirty）则继续 `RequestVsyncFrame()` 自驱动。
- `OnFrameTick()`：**只保留 resize 检测**，不再渲染。
- 新增 `std::atomic<bool> m_framePresented`：每次实际 flush 置 true。
- 新增 `DrainPendingFrame()` + NAPI 导出 `drainPendingFrame`（注册进 `desc[]`）。
- 生命周期：`OnSurfaceDestroyed` / 析构函数中，在锁 `m_surfaceMutex` **之前**调用 `DestroyVsync()`（避免与 `OnVsyncTick` 持锁死锁）；`DestroyVsync` 先置空指针、再重置 `m_vsyncPending`。

**CMake (`CMakeLists.txt`)**

- `find_library(native-vsync-lib native_vsync)` 并链接。

**ArkTS (`TerminalController.ets`)**

- 桥接接口加 `drainPendingFrame: () => boolean` + 同名 passthrough 方法。

**ArkTS (`TerminalSurface.ets`)**

- 新增 `@State private framePresentTick: number` + `framePresentTimer`。
- build() Stack 内加一个不可见 1px `Text(`${this.framePresentTick % 2}`)`（opacity 0.01），保证窗口帧不被优化掉。
- `startFramePolling()`：16ms `setInterval`，`drainPendingFrame()` 为真则 `framePresentTick++`。
- 在 `onLoad`（渲染器就绪时）启动，`onDestroy` / `aboutToDisappear` 停止；fallback 文本模式下不启动。

## 7. 关键技术点速查

- **XComponent SURFACE 模式**：App 自己拿 NativeWindow 渲染；surface 图层的 RS 合成由 ArkUI 窗口帧门控。
- **OnFrameTick**（`OH_NativeXComponent_RegisterOnFrameCallback`）：在 ArkUI 帧管线内、UI 线程触发；**ArkUI 空闲时会停止**。
- **OH_NativeVSync**：独立的硬件 vsync 连接，空闲时也持续触发；能驱动绘制，但**不能**唤醒窗口合成。
- **OnVsyncTimeOut**（WMSMain）："ArkUI request vsync, but no vsync received" → 窗口合成管线休眠的标志。
- **唤醒窗口合成的唯一可靠途径**：让 ArkUI 提交窗口帧（按键、触摸、或**改变 @State 重跑 build**）。
- **锁约定**：`m_renderer` / `m_terminal` / `m_surfaceReady` / `m_lastBlinkMs` 受 `m_surfaceMutex` 保护；`m_renderDirty` / `m_trailingFrames` / resize 标志受 `m_renderMutex` 保护。
- **acquire fence**：`RequestBuffer` 返回的是 acquire fence（消费者读完才可写），需 poll 等待后 close；**不可**当作 release fence 传回 `FlushBuffer`，否则会 stall BufferQueue 并触发 OnVsyncTimeOut。

## 8. 已知历史修复（相关背景）

- 光标不闪烁：之前 session 由 commit `dd04872` 恢复解决。
- 多标签页 IME 攻击：进程级单一 editor 附着，用 `g_activeImeHost` 跟踪当前所有者，避免后台标签每帧推送失效 proxy。
- 标签关闭冻结：IME detach 必须在 `m_surfaceMutex` 之外做（detach 会同步回调我们的 editor 回调，二者都锁该非递归 mutex → 自死锁）。
- 字体集合：`OH_Drawing_CreateSharedFontCollection()` 为进程级共享单例，单个 renderer 析构时**不可**销毁。
